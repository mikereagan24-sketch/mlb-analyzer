## Section 1 — Confirmed bugs (with evidence)

### 1.1 HIGH — PR #142 over-neutralizes Steamer projections (my new bug)

**Pattern class:** 4 (double count via the fix itself — the case the brief flagged as "plausible new bug").

**Evidence:** Sampled projected wOBA for 30–40 hitters per team from `woba_data.bat-proj-rhp` (Steamer RoS, vs RHP), grouped by team suffix:

| Team | Park factor | Projected mean wOBA | Δ vs league-avg (LAD/TOR/HOU mean 0.2983) |
|------|-------------|--------------------:|------------------------------------------:|
| COL  | 1.10        | 0.3017              | **+1.16%** (would be +5–10% if park-adjusted) |
| ATH  | 1.09        | 0.3027              | +1.47% |
| CIN  | 1.05        | 0.3019              | +1.19% |
| LAD  | 1.00        | 0.3036              | +1.79% (higher than COL!) |
| SEA  | 0.96        | 0.3010              | +0.92% |
| SD   | 0.96        | 0.2979              | −0.12% |
| SF   | 0.94        | 0.2964              | −0.63% |

The variance across teams is roster-quality noise (±1.8%); there is no correlation with park factor. Steamer projections are **park-neutral**.

**Impact:** In `services/model.js` `neutralizedOrRaw` (added by PR #142), when `PARK_NEUTRAL_INPUTS_ENABLED` is ON, the projection component and actuals component are BOTH deflated by `1 / (1 + (factor − 1) / 2)`. But only the actuals side carries a home-park signature; the projection side does not. Under the current blend `W_PROJ=0.45, W_ACT=0.55` (verified from `getSettings()` output during backtest run), the correct deflation for a COL hitter is `0% × 0.45 + 4.8% × 0.55 = 2.64%`; the code applies `4.8% × 1.00 = 4.8%`. **Over-neutralization of ~2.2 percentage points on extreme-park hitters.**

The PR #142 backtest still showed net positive (aggregate totals +3.32pp ROI, affected +4.39pp) because the raw double-count of both components was a larger error than the current over-correction. But the "correct" fix would help more.

**Bounded fix:** Apply neutralization only to the actuals-side value BEFORE `blendWoba` combines it with the projection value. Or, less invasive, apply post-blend but scale by the actuals fraction: `neutral = raw × (W_PROJ + W_ACT / (1 + (factor − 1) / 2))`.

**Verification:** Re-run `scripts/backtest-park-neutral.js` after the correction and compare against current PR #142 ON results. Expected: further ROI improvement on affected-team totals (~1–2pp), same or better on ML.

**File evidence:** `services/model.js:141–148` (`neutralizedOrRaw`), `:130–139` (called in `getBatterWoba`), `:184–187` (called in `getPitcherWoba`); `services/park-factors-woba.js:81–101`.

### 1.2 HIGH — Backwards SP-weight fallback: missing forecast → max confidence (0.80)

**Pattern class:** 3 (backwards fallback — precedent revisit; still present).

**Evidence:** `services/model.js:1119`:
```js
function computeSpPitWeightFromForecast(forecastIp, settings, nPriors) {
  if (forecastIp == null) return null;   // ← returns null when no forecast at all
  ...
```
Called at `services/model.js:670–671`:
```js
const awaySpPitW = computeSpPitWeightFromForecast(awayFc, settings, awayNP) ?? SP_PIT_WEIGHT;
const homeSpPitW = computeSpPitWeightFromForecast(homeFc, settings, homeNP) ?? SP_PIT_WEIGHT;
```
`SP_PIT_WEIGHT` default is 0.80 (`services/model.js:353`, `settings-schema.js:15`).

**The problem:** Zero priors (function ran, found no history) → `SP_FORECAST_LOW_CONF_TARGET` = **0.62** (conservative, correct). No forecast at all (pitcher entirely absent from the forecast index) → `null` → falls through `??` to **0.80** (max confidence). The two "no data" cases are inverted: less information should produce lower weight, not maximum weight.

The code comment at `services/model.js:665–667` calls this out explicitly as "preserving prior behavior for games with no forecast" — so it's intentional back-compat, but the pattern is exactly the precedent bug: missing data → maximum confidence.

**Impact estimate:** Pitchers with zero forecast are rare (usually rookies / mid-season call-ups). Whenever they hit an SP slot the model over-weights their contribution vs bullpen by ~29% relative (0.80 vs 0.62). If the pitcher is unproven, this systematically overrates the whole SP side.

**Bounded fix:** Change `services/model.js:1119` from `return null` to `return targetUsed`. Also update the comment at 665–667 to reflect the new semantic. Regression check: a pitcher with a forecast + full priors path must yield identical weight to today.

**Verification:** Enumerate all no-forecast games in the backtest window (`SELECT ... WHERE away_sp_forecast_ip IS NULL OR home_sp_forecast_ip IS NULL`); rescore with the fix; compare accuracy and CLV on that population.

### 1.3 HIGH — Totals systematically over-projected on OVERS in mid-range totals (7–9)

**Pattern class:** targeted A investigation; measurement-backed.

**Evidence** (from live DB slice by graded signals):

| Side | n | Wins | Losses | Win rate | ROI | PnL |
|------|--:|-----:|-------:|---------:|----:|----:|
| Over | 75 | 35 | 40 | 46.6% | −861% (÷100 = −8.6%) | −646 |
| Under | 143 | 73 | 70 | 50.8% | −106% (÷100 = −1.1%) | −152 |

Mid-range subset (market total 7–9):

| Side | n | Avg market | Avg model | Wins | Losses | ROI |
|------|--:|-----------:|----------:|-----:|-------:|----:|
| Over | 60 | 7.9 | **8.7** | 25 | 34 | −1615% (÷100) |
| Under | 87 | 8.3 | **7.3** | 38 | 44 | −894% (÷100) |

**The model projects ~0.8 runs higher than market on average for mid-range totals when it fires an OVER**, and ~1.0 run lower when it fires an UNDER. The magnitude of gap doesn't correspond to real edge — the OVER edges lose worst.

**Comparison to ML:** ML: 386 signals, +134% ROI, 43.5% win rate. Totals: 218 signals, −366% ROI, 45.4% win rate. **Totals are the model's weakest leg by ~18 percentage points of ROI.**

**Suggested investigation (not fix yet):** Two candidate mechanisms:
- **TOT_SLOPE too aggressive at mid-range.** `services/model.js:876–878` uses linear `modelOverP = 0.5 + runDiff × TOT_SLOPE`. With `TOT_SLOPE=0.08`, a 1-run gap becomes 8pp of edge; empirical distribution in the 7–9 total range is likely tighter (fewer edges of that size actually materialize).
- **Overs benefit from something the model captures wrong.** Weather run adjustments (`services/model.js:759–762`) apply on top of park factor. If wind/temp priors are too bullish on scoring, they compound the projection into false OVER edges.

**Bounded next step:** Backtest with `TOT_SLOPE=0.06` on games where `7 ≤ market_total ≤ 9`, over side only. Report Δ ROI + volume.

### 1.4 HIGH — KC ML bleed concentrated on home-underdog in Apr–May, edge magnitudes implausible

**Pattern class:** targeted B investigation; measurement-backed.

**Evidence:**
- Total: 35 signals, 14–19 W/L, ROI −1406% (÷100 = −14%), pnl −492.
- **Home vs road:** home 19 signals ROI −2769%, road 16 signals ROI +214%. **Home is the entire bleed.**
- **Fav vs dog:** dog 21 signals ROI −1980%, fav 14 signals ROI −543%. **Dog is the sharper edge case.**
- **Home + dog intersection:** 13 signals, 4–9 W/L, ROI −2924%, pnl −380.
- **Temporal:** 32 of 35 signals (91%) fell in April–May 2026; June + July combined = 3 signals.
- **KC ML edge distribution:** mean 22.0pp. Individual bets show 40pp+ edges (e.g., 2026-05-19 bos-kc: edge 40pp, loss; 2026-05-16 kc-stl: edge 40pp, loss). League ML edges are typically 2–5pp for signals that grade at ~50%.

**Interpretation:** The model was massively over-confident on KC as a home underdog early season. 40pp+ edges are implausible against real closing lines (they should have been arbitrage-worthy if real). The concentration in April/May and near-vanishing after early June points to *early-season input calibration* rather than an ongoing bug — but the mechanism isn't fully identified.

**Candidates to investigate (not this pass):**
- KC wOBA input calibration relative to actuals in April/May
- Roster instability (traded / promoted players) sending too many KC batters to `BAT_DFLT_START` fallback
- KC bullpen wOBA blend using stale or thin data in early season

**Bounded next step:** For each of the 35 KC signals, pull the model's `aTeamWoba` and `hTeamWoba` snapshots (if persisted) and compare against what would be reconstructed from *today's* woba_data with the same lineup. Systematic overrating on the KC side is the tell.

### 1.5 MEDIUM — Framing catcher resolution: first-pos-C picker, no disambiguation

**Pattern class:** 3 (backwards fallback) + targeted D.

**Evidence:** `services/jobs.js:413–417`:
```js
const findCatcher = (lu) => {
  const arr = tryParse(lu) || [];
  const c = arr.find(p => (p.pos || '').toUpperCase() === 'C');
  return c ? c.name : '';
};
```
This is a first-match picker: whichever player has `pos === 'C'` first in the array wins. No cross-check against team_rosters' actual catchers, no fallback to batting order (catchers historically bat 8th or 9th), no disambiguation when two players are tagged 'C'.

The precedent log (`team=MIL lineup_name="Luis Rengifo"`) implies either (a) upstream lineup JSON assigned pos='C' to a non-catcher, or (b) two rows were tagged 'C' and the wrong one won. The upstream parser at `services/scraper.js:357–370` (RotoWire parse) assigns `pos = parts[0]` from strings like `"RF Carson Benge L"` — if RotoWire ever mis-orders the tokens, the pos is wrong at ingest and never re-checked at read.

**Impact estimate:** Framing delta is muted (default `CATCHER_FRAMING_MUTE=0.5`) and gated (`CATCHER_FRAMING_ENABLED=true`), so a wrong catcher's framing rating substitutes silently for the real one. Magnitude: at most ±0.15 runs per game per side (typical framing signal size × mute), so per-game direct impact is small — but it moves the totals line by the same amount, which matters at the ROI margin given totals are already the weak leg (finding 1.3).

**Bounded fix (not this pass):** Add a second-pass sanity check: if first-pos-C match doesn't appear in `team_rosters` as a catcher by mlb_id, walk the lineup again looking for the lowest batting-order player who does appear in the roster's catcher list; if that also fails, return null (no framing applied) rather than substituting the wrong catcher.

**Verification:** Enumerate last 30 days of resolved games; for each, compute framing catcher via current logic and via the sanity-checked logic; count divergences. If <2%, the current path is safe enough. If >5%, this is a real bias worth fixing.

### 1.6 MEDIUM — Local DB gap 2026-06-15/16 is a LOCAL artifact, not a production job failure

**Pattern class:** targeted C investigation.

**Evidence:** Production `/api/health/2026-06-15` returns `games_total: 10`; `/api/health/2026-06-16` returns `games_total: 15`. Both dates have `sp_coverage: pass`, `ml_coverage: pass`, `totals_primary: pass`. **Production has data for both dates.** The gap in the local DB the brief flagged reflects the local mirror not being pulled on those dates, not a scoring / ingest failure.

**Backfill status:** No backfill needed — data is already in prod. If the user wants the local DB current, pull it via the existing `/admin/download-db` route.

**Adjacent finding — no root-cause for job gaps if they DID occur:** The ingest jobs' timezone logic uses PT-anchored dates (`services/jobs.js:19–26`), but the MLB API expects ET boundaries. A 10:30pm PT game on June 15 = 1:30am ET June 16. If `runScoreJob` fires at midnight PT June 16 with `yesterdayStr()` = "2026-06-15", the fetch is for ET June 15 — games from the following ET window aren't included. This is a **latent risk pattern**, not the cause of the actual gap that turned out to be local-mirror-only. Worth documenting but not the source of the 06-15/16 issue.

### 1.7 MEDIUM — Health-check coverage gaps: framing, FRV, baserunning, roster jobs

**Pattern class:** 2 (silent non-fatal failures).

**Evidence:** `/api/health/:date` `cron_health` check reads `cron_log` and specifically looks for entries with `job IN ('odds', 'lineups', 'scores')` (verified at `routes/api.js` around the `cron_health` check ID). The 6 AM PT block chains six additional jobs internally:
- `runRosterJob`
- `runSeasonRosterJob`
- `runFieldingFrvJob`
- `runBaserunningJob`
- `runPlayerBaserunningJob`
- `runPlayerBaserunningTrailingJob`

Each is wrapped in try/catch with `console.error` only. Failures don't write to `cron_log` (they're inside the wrapping cron handler, not separate entry points), so `cron_health` reports "pass" even if all six failed. The `pitcher_fg_role` freshness check exists (from PR #140), so roles have coverage; **framing, FRV, and baserunning freshness checks do NOT exist**. If the framing ingest fails silently for a week, no dashboard signal.

**Impact:** Model quietly falls back to null framing / null FRV / null BsR on affected games; the model still fires signals but with a wrong (default) run environment. The precedent for this exact class was the fg-roles 403 that ran silently for a month.

**Bounded fix (not this pass):** Add three new health checks: `framing_fresh`, `frv_fresh`, `baserunning_fresh`, each with the same 3d/7d threshold pattern used for `fg_roles_fresh`. Requires a `MAX(updated_at)` query per input table and a message that names the fix ("run the 6 AM PT job block" or the specific admin refresh route).

## Section 2 — Probable biases needing measurement

### 2.1 Opener mis-routing: `is_opener_game_*` gate is defended, but `was_starter` tagging may still send openers through the SP path when the FG confirming signal fails

**Pattern class:** 3 (backwards fallback via a two-signal gate).

Openers are detected via (a) FG RosterResource role classification AND (b) recent pitcher_game_log `avg pitches < 30` over last 10 outings (`services/jobs.js` opener detection block). When either fails, the pitcher runs the standard SP path in `runModel`, which invokes `forecastSpIP` — a function anchored to 5.5 IP as league default. A true opener whose FG tag says SP but who consistently throws 1 IP would get an inflated SP forecast and correspondingly inflated `SP_PIT_WEIGHT` (approaching 0.80).

**Measurement needed:** Enumerate pitchers with average `pitches_thrown` < 30 across their last 10 outings AND `was_starter=1` in `pitcher_game_log`. For each, check whether `is_opener_game_*` was set on any recent game. If pitchers matching the profile exist without opener flags, that's the confirmation.

**Priority:** Medium. Impact is bounded (rare edge case), but a whole SP-weight class getting the wrong end of a 0.62 vs 0.80 swing is a clear source of noise.

### 2.2 Steamer projection age / vintage

Projections are `rsteamer_vl_*` — Rest-of-Season Steamer. Fetched via the FG API which returns the *current published* RoS. But: is Steamer's RoS regenerated daily by FG, or weekly? If it lags 3–7 days, the recent-form component is stale. The bookmarklet pulls fresh data every day the owner runs it, so downstream freshness == upload cadence. No health check flags stale projections (the `bat-proj-*` freshness is `uploadedAt`, not "Steamer's own compute date").

**Measurement needed:** Compare a player's current wOBA projection to their projected wOBA of a month ago (via `woba_data_snapshot` if enabled, else compare live upload times). If projection changes are <0.005 wOBA points month over month for a healthy player, Steamer isn't refreshing aggressively — recent hot/cold streaks are absent from the input. Not a bug; a bias worth quantifying.

---

### 2.3 Bullpen wOBA blend recency

`services/model.js:509–513` reads per-handedness bullpen wOBA from `game.awayBullpenVsL / awayBullpenVsR` etc., built server-side by `getBullpenWobaBlended` (`db/schema.js`). The blend rolls up available RPs on the team. If a team's rotation shift changes bullpen roles (e.g., closer moves to setup), the roll-up doesn't know about the role change until FG Daily Sync roles refresh. **Bullpen composition can lag reality by days.**

**Measurement needed:** Compare bullpen wOBA blends before vs. after a role change (e.g., someone flipped from CL to MID); quantify the delta and how many games it affected.

**Priority:** Low. Bullpen wOBA moves in tenths of a run per side; hard to see in signal-level ROI without a large sample.

`services/kalshi.js:281`: `KALSHI_FEE_COEF = 0.068`. Comment at `~262–280` documents that this was back-solved from live fee slips at `C=0.62`; the recurring decimal `2.56/44 = 0.05818…` was the corrected form found during Kalshi order-book work. ## Section 3 — Fragile-but-working

### 3.1 Steamer season-anchored constants: `WOBA_BASELINE=0.230`, `PYTH_EXP=1.83`, `RUN_MULT=48`, `HFA_BOOST=0.02`

All at `services/model.js:353–356`. Provenance comments are thin:
- `WOBA_BASELINE`: no comment on which season it was calibrated to. Modern MLB is `~0.230-0.235`. A shift to 0.235 would systematically bias every team's run projection downward by ~0.5 runs.
- `PYTH_EXP`: 1.83 — standard for MLB, unlikely to move but no cite.
- `RUN_MULT`: 48 — no comment on origin. Empirically reasonable but no re-check gate.
- `HFA_BOOST`: 0.02 — no comment on data window. Historical HFA is ~54% at home; 0.02 boost implies model's baseline WP already includes ~52%.

**Not bugs**, but any of these could rot over 2–3 seasons and no mechanism flags it. Suggested: add a per-season calibration gate (e.g., a Jan 1 health check that compares current constants against a rolling empirical estimate and warns if they diverge > 5%).

### 3.2 Two park-factor tables coexisting

PR #142 added `services/park-factors-woba.js` (wOBA-scale) alongside the pre-existing `services/scraper.js` `PARK_FACTORS` (run-scale). Both work correctly today, but the potential for a future caller to import the wrong one is real. Current callers audited:
- `services/scraper.js` PARK_FACTORS: read only in `resolveParkFactor` (scraper.js) and via `pickVenueOverride` (model.js:679) → applied at model.js:686 as `pf`. Run-scale, correct.
- `services/park-factors-woba.js` WOBA_PARK_FACTORS: read only via `getWobaParkFactor` in the neutralization path. wOBA-scale, correct.

Suggested: at minimum, add a comment header on each table explicitly cross-referencing the other so a future reader knows why there are two.

### 3.3 `blendWoba` min-sample gate is binary

`services/model.js:69–77`: `blendWoba(proj, act, minSample, wProj, wAct)` gates the actuals row on `act.sample >= minSample`. Below the floor, actuals are dropped and projection wins outright. This is fine for real "noise" cases (5 PA cameo) but wrong for legitimate low-sample signals (a rookie with 55 PA of a hot start, or a pitcher with 90 BF of collapse). Filter lacks a pattern-vs-fluke distinction.

**Not a confirmed bias yet** — need to measure how many signals in the current graded set had their actuals silently dropped. But it's a filter that can erase systematic patterns, matching the pattern-5 template.

### 3.4 Framing / FRV inputs are muted by hard defaults, not by uncertainty

`CATCHER_FRAMING_MUTE=0.5`, `DEFENSE_FRV_MUTE=0.5`: both hardcode a "half apply, half don't" that assumes framing/defense already partially embed in wOBA-against. The 0.5 is arbitrary — the correct mute is measured, not assumed. Right now the model doesn't know how much of the framing signal is redundant with what's already in pitcher wOBA-against for that batterymate.

**Recommendation:** the muting parameter is a top-of-list tuning target (see Section 4).
